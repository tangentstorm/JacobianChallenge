* Root Cause: IntegralOneCycle Specialization

Location: Jacobian/Periods/IntegralOneCycle.lean:28

The definition:
  noncomputable def IntegralOneCycle (X : Type) [TopologicalSpace X] :
      ModuleCat ℤ :=
    ((AlgebraicTopology.singularHomologyFunctor (ModuleCat ℤ) 1).obj
      (ModuleCat.of ℤ ℤ)).obj (TopCat.of X)

Forces (X : Type 0) because:
- singularHomologyFunctor requires [HasCoproducts.{w} (ModuleCat ℤ)] where w matches X's universe
- Mathlib v4.28.0 provides HasCoproducts.{0} (ModuleCat ℤ) only
- No universe-polymorphic instance HasCoproducts.{u} (ModuleCat ℤ) exists in Mathlib

The file itself documents this (lines 21-28):
  "Note on universe: explicitly threaded as universe 0 (X : Type) because
   singularHomologyFunctor requires [HasCoproducts.{w} (ModuleCat ℤ)]
   where w matches X's universe, and the available instance for
   ModuleCat ℤ is at universe 0."

**This is a REAL Mathlib constraint, not a project-internal gap.**

* Degenerate ULift Analysis

In Solution.lean lines 136–137, the definition uses:
  ULift (JacobianChallenge.ComplexTorus.quotient
    (Fin (genus X) -> ℂ) (JacobianChallenge.Periods.periodFullComplexLattice X))

Given:
- periodFullComplexLattice X is Type 0
- quotient (Fin (genus X) -> ℂ) (periodFullComplexLattice X) : Type 0

The ULift here is ULift.{0, 0} : Type 0 -> Type 0, which is DEGENERATE.

The docstring (line 131) documents this:
  "The ULift in the body is now degenerate (ULift.{0,0} is ULift to the same universe) 
   but kept for shape parity with the universe-poly intent."

Non-degenerate intent: once periodFullComplexLattice X is lifted to (X : Type u),
the ULift becomes ULift.{u, 0}, transporting the Type 0 quotient up to Type u.

* Conclusion

The universe specialization from (X : Type u) to (X : Type 0) in Solution.Jacobian 
is driven by TWO REAL MATHLIB CONSTRAINTS:

1. **IntegralOneCycle** – mandated by singularHomologyFunctor's dependence on 
   HasCoproducts.{0} (ModuleCat ℤ) in Mathlib v4.28.0

2. **periodPairing** – inherits (X : Type) from IntegralOneCycle in its codomain

These are NOT spurious project-internal gaps that can be fixed by adding 
universe annotations.

**Lifting Solution.Jacobian to universe-polymorphic status requires:**
- Upstream Mathlib instance: HasCoproducts.{u} (ModuleCat ℤ), OR
- Internal ULift bridging with careful propagation, OR
- Accept specialization (current design choice)

The project has chosen specialization at the data level while maintaining 
theorem-level compatibility with Challenge.Jacobian.
